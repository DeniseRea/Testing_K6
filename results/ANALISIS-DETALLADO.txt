2. DESCRIPCIÓN DE MÉTRICAS
================================================================================

VUs (Usuarios Virtuales):
  - Número de usuarios simultáneos que acceden al sistema durante la prueba
  - Valores probados: 100, 150, 200, 300

Avg Duration (ms):
  - Tiempo promedio de respuesta de las solicitudes HTTP
  - Métrica clave para medir la latencia del sistema

P(95) - Percentil 95 (ms):
  - El 95% de las solicitudes se completaron en este tiempo
  - Importante para identificar picos de latencia

Failed Requests:
  - Número de solicitudes que fallaron durante la prueba
  - Incluye timeouts y errores de conexión

Total Requests:
  - Número total de solicitudes HTTP completadas

5. CONCLUSIONES Y RECOMENDACIONES

5.1 HALLAZGOS CLAVE:
  ✓ Sistema estable hasta 200 VUs
  ⚠ Punto crítico identificado entre 150-200 VUs
  ❌ Límite absoluto en 300 VUs con fallos evidentes
  
  ✓ Anomalía: Mejor performance en 200 VUs que en 150 VUs
    → Sugiere auto-scaling o equilibrio de carga dinámico

5.2 RECOMENDACIONES OPERATIVAS:

  CORTO PLAZO:
  - Mantener carga máxima en 200 VUs
  - Monitorear métricas en tiempo real
  - Implementar alertas en P(95) > 500ms
  
  MEDIANO PLAZO:
  - Investigar degradación anómala en 150 VUs
  - Optimizar consultas de BD
  - Aumentar pool de conexiones
  - Implementar caché (Redis/Memcached)
  
  LARGO PLAZO:
  - Arquitectura de microservicios
  - Escalado horizontal automático
  - CDN para contenido estático
  - Load balancing distribuido geográficamente

5.3 UMBRAL DE ACEPTABILIDAD:
  
  Operación Normal (≤100 VUs):
  - P(95) < 500ms ✓
  - Tasa de error < 0.1% ✓
  - Recomendado para producción
  
  Operación Aceptable (101-200 VUs):
  - P(95) < 600ms ✓
  - Tasa de error < 0.1% ✓
  - Requiere monitoreo
  
  Operación Crítica (201-300 VUs):
  - P(95) > 700ms ✓
  - Tasa de error ≥ 0.1% ⚠️
  - Requiere escalado inmediato
  
  Sobrecarga (>300 VUs):
  - P(95) > 1000ms ❌
  - Tasa de error > 1% ❌
  - No recomendado sin optimización
